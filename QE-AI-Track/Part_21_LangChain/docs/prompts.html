§1Build timeline

The order in which the pieces came together, driven by the conversation.

  • Switch Groq → OpenRouter (Level 1)Move the bare LLM call onto OpenRouter (ChatOpenAI + base_url), model from .env (DeepSeek).
  • Fix the v1 agent importcreate_tool_calling_agent + AgentExecutor are gone from langchain.agents in v1 → switch to create_agent, invoked with {"messages":[…]}.
  • Bug triage agentfetch_jira_ticket tool (JIRA REST v3 + ADF→text) → structured Triage. Fixed the Groq string-boolean 400 with method="json_schema".
  • RAG knowledge baserag/ package — Chroma KB (Ollama nomic-embed-text), ingests test cases + bug reports + PDFs + live JIRA (JQL). Prod triage grounds each verdict in the top-3 similar items.
  • RCA agent (demo → production)Root-cause agent with real tools: Jenkins console log, GitHub commits, live health ping. Fixed an LLM hallucinating the health URL by removing the tool argument.
  • RCA Streamlit UIsrc/UI/rca_ui.py — enter Jenkins / GitHub / app URL / question, run the agent, see the evidence trail + RCA.
  • Test designer (demo → production)Requirement → structured TestPlan. Production version fetches a JIRA story (project VWO) and pulls context from the QA Copilot RAG before designing — ran on VWO-105 → 18 grounded cases.
  • Architecture diagramdocs/test_designer_architecture.html — Excalidraw-style pipeline of the test designer.

§3Bug triage agent

src/bug_traige_agent.py — JIRA → structured Triage

tools: fetch_jira_ticket · output: Triage(title, severity, priority, component, suggested_team, likely_duplicate, reasoning)
JIRA REST v3ADF→textjson_schemamulti-ticket
System prompt
You triage bugs for a web product. Map user impact to severity
and business urgency to priority. Be decisive.
Human prompt
Bug report:

{report}

src/bug_traige_agent_prod_ready.py — RAG-grounded triage

adds top-3 retrieval from the QA Copilot RAG before triaging (reuses precedent, avoids dupes)
Chroma KBOllama nomic-embedprecedent-grounded
Structured-output gotcha (Groq): default function-calling lets the model emit "False" (a string) for a boolean field → Groq rejects it (400 tool_use_failed). Fix: with_structured_output(Triage, method="json_schema") with a json-schema-capable model (e.g. openai/gpt-oss-120b; llama-3.1-8b-instant does not support it).

§4Root-cause analysis agent

src/rca_agent_prod_jenkins_github_apitestping.py

tools: fetch_test_logs (Jenkins console) · recent_commits (GitHub API) · service_health (live ping)
JenkinsGitHub commitshealth pingcreate_agent
System prompt
You are a root-cause analysis agent for test automation. Gather evidence
with the tools BEFORE concluding — check the Jenkins log, the recent commits,
and whether the app under test is up. Then answer: is the automation healthy?
Is there a login error? Give the single most likely root cause, the evidence
for it, and one concrete fix. If a tool returns an ERROR, say so and reason
from what you do have.
Example user input
Can you check the RCA — is the automation working fine,
and do we have any error in the login?
Fixed: the LLM invented a fake health URL (ttacart.example.com) → false "DOWN". Removing the url argument from service_health() forces it to always ping the configured app. Also has a Streamlit UI: src/UI/rca_ui.py.

§5Test designer agent

src/test_designer_agent_prod.py — JIRA story → RAG-grounded TestPlan

tools: fetch_jira_story (project VWO) · qa_copilot_search (RAG) · output: TestPlan(feature, scope, risks, test_cases[])
JIRA VWOQA Copilot RAGstructured TestPlan
System prompt
You are a test design expert. Turn a requirement into a crisp plan and
concrete, runnable cases. Cover happy path, negative, edge and security.
Reuse the conventions (ids, style, teams) shown in the PRIOR TEST CASES and
avoid duplicating cases that already exist there. No fluff.
Human prompt
Requirement / user story:

{story}

PRIOR TEST CASES from the QA Copilot knowledge base:

{prior}
Result — VWO-105 (SALCart PRD)
Feature: SALCart End-to-End Functional, Negative, Edge & Security Test Suite (18 cases)
Cases: TC-SALC-001 … TC-SALC-018
Covers: login, inventory/sort, cart, both checkout steps, hamburger menu,
        responsive layout, performance_glitch_user, security of protected pages
Risks:  regression of core flows · wrong cart/tax total · auth bypass via direct URL
